# AI Agent 可持续编程与交接规则

日期：2026-06-09

## 1. 规则目标

本文件用于保证多个 AI Agent 可以连续接手项目，而不会覆盖他人代码、扩大范围、跳过测试或错误宣称完成。

本文件属于项目强制规则。任何 Agent 修改代码前必须阅读。

## 2. 核心执行模型

所有工作必须使用以下最小闭环：

```text
读取规则与现状
-> 领取一个任务包
-> 建立可复现失败或 RED 测试
-> 实现最小变更
-> Review 与修复
-> 验证门禁
-> 独立 Git 提交
-> 标准交接
```

一个 Agent 一次只能领取一个可独立验收的任务包。任务包通常应在一个模块内完成，禁止一次接管整个剩余项目。

## 3. 工作区所有权规则

### 3.1 遇到未提交变更

任何未提交变更默认属于用户或其他 Agent。

接手 Agent 必须：

1. 运行 `git status --short --branch` 和 `git diff --stat`。
2. 列出与自己任务相关和无关的变更。
3. 只在任务明确要求且理解现有变更后继续修改相关文件。
4. 不暂存、不提交、不格式化无关文件。

禁止：

- 使用 `git reset --hard`、`git clean -fd`、`git checkout -- .`。
- 删除无法理解的文件以恢复构建。
- 用脚手架重新生成并覆盖已有模块。
- 把其他 Agent 的未提交变更包含进自己的提交。

### 3.2 当前特殊保护范围

`frontend/package.json` 和 `frontend/package-lock.json` 已确认是重复根依赖文件。按 D-015，只允许在独立的 `10F-R5` 清理任务中验证后删除；其他任务仍不得顺带修改或删除。

后续 Agent 必须先按 `docs/23-phase-10f-review-fix-and-10g-quality-plan.md` 执行阶段 10F-R，不允许直接跳到质量基建、小程序、后端补缺或发布阶段。

## 4. 任务包规范

每个任务包开始前必须明确：

```text
任务包 ID：
目标：
允许修改文件：
禁止修改文件：
依赖接口或前置阶段：
已知阻塞：
RED 测试或失败证据：
完成验证命令：
预期提交信息：
```

任务包范围标准：

- 应能在一次 Review 中理解。
- 应能通过一组明确命令验证。
- 应能形成一个或少量紧密相关的提交。
- 不应同时修改小程序、管理后台、后端和数据库。
- 如果需要跨层修改，必须拆成“后端契约提交”和“前端接入提交”。

## 5. 阶段和依赖规则

- 严格按 `docs/21-remaining-development-roadmap.md` 的依赖顺序执行。
- 当前阶段未通过退出门禁，禁止开始依赖它的下一阶段。
- 可以并行的任务必须文件范围互不重叠，并有独立验证路径。
- 发现计划与真实代码不一致时，先更新规划文档或交接记录，再实现。
- 未决事项只能分析和给方案，不能自行实现。

## 6. 编码规则

### 6.1 通用规则

- 只实现当前任务包要求。
- 优先使用现有架构和工具，不引入重复框架。
- 避免 `any`、魔法字符串、深层嵌套和超长函数。
- 不做与任务无关的格式化、重命名和重构。
- 所有外部输入必须校验。
- 错误必须显式处理，不能静默吞掉。

### 6.2 前端规则

- 页面只能通过统一 API Client 访问后端。
- API 类型必须来自真实 Controller、DTO 或接口清单。
- 页面不能伪造成功结果。
- 路由、菜单和按钮可按权限隐藏，但后端仍是最终授权方。
- 每个页面必须覆盖 Loading、Empty、Error、Forbidden 和 Success 状态。
- 写操作必须防重复提交，高影响动作必须二次确认。
- 不使用 `v-html` 渲染用户或 AI 内容，除非有可信净化。
- 只有跨页面共享状态才放入 Pinia。
- 状态显示使用集中状态字典，不在页面散落字符串。

### 6.3 后端规则

- Controller 不直接调用 Mapper。
- DTO 不直接暴露实体。
- 资源归属校验不能由前端参数决定。
- 多表状态变更必须使用事务。
- API、状态、权限码或 Schema 变更必须先更新契约和影响说明。
- 不为了前端联调放宽认证、权限或业务校验。

## 7. 测试驱动规则

非平凡逻辑必须执行：

1. RED：新增测试或记录可复现失败。
2. GREEN：实现最小代码使测试通过。
3. IMPROVE：在测试保持绿色时重构。

测试要求：

- 修改请求层、认证、权限、状态流转、金额、内容审核或 AI 安全边界时必须有直接测试。
- 修复 Bug 时必须先增加能复现该 Bug 的回归测试。
- 不允许删除失败测试来通过门禁。
- 不允许降低断言强度、跳过测试或把真实失败改成假成功。
- 业务逻辑覆盖率目标不低于 80%，关键规则不以覆盖率代替直接测试。

## 8. Review 规则

代码完成后必须进行 Review，优先级如下：

1. 正确性和业务规则。
2. 认证、权限、资源归属和秘密泄漏。
3. 数据一致性、事务和并发。
4. 错误处理和失败状态。
5. 测试有效性。
6. 可维护性和可读性。

CRITICAL 或 HIGH 问题未解决前，不允许提交阶段完成结果。

## 9. 验证规则

### 9.1 前端最低验证

```powershell
npm run lint
npm run typecheck
npm run test
npm run test:coverage
npm run build
```

关键流程还必须运行对应 E2E。命令尚未建立时，应先完成阶段 10G，不能长期以“没有脚本”为理由跳过验证。

### 9.2 后端最低验证

```powershell
mvn test
mvn package "-DskipTests"
```

涉及真实 MySQL 并发、Schema 或 E2E 时，必须运行对应专项验证。

### 9.3 所有任务通用验证

```powershell
git diff --check
git status --short --branch
```

没有运行的验证必须明确写为“未验证”，并说明原因和风险。

## 10. Git 持续开发规则

- 不直接在 `main` 开发。
- 使用路线图指定阶段分支。
- 每个提交只包含一个可解释目的。
- 使用 Conventional Commits。
- 提交前查看完整 `git diff`。
- 使用路径限定的 `git add -- <paths>`，避免暂存其他 Agent 的文件。
- 当前工作区有无关未提交变更时，提交后必须再次确认它们仍然存在且未被纳入。
- 禁止强制推送、重写历史和删除分支，除非用户明确授权。

推荐提交粒度：

```text
test(admin-web): add request error mapping tests
feat(admin-web): add request error mapping
feat(admin-web): add booking management page
test(user): add pet ownership integration tests
feat(user): add pet profile API
docs: update phase handoff
```

## 11. 接口契约变更规则

当 Agent 发现前端需要后端新接口或字段时：

1. 先确认真实 Controller、DTO 和已有文档中确实不存在。
2. 记录用途、请求、响应、权限码、错误和兼容性影响。
3. 将后端实现拆成独立任务包。
4. 后端测试与提交完成后，前端才能接入。
5. 禁止前端临时猜测字段后继续扩散。

## 12. 阻塞与恢复规则

遇到阻塞时：

- 保留可复现命令和错误摘要。
- 标记被阻塞任务包，不把整个项目错误标记为完成。
- 继续执行同阶段中无依赖的任务。
- 连续三次无法推进且必须依赖用户决策时，才请求用户输入。

发现前一 Agent 的代码错误时：

- 先新增回归测试或复现证据。
- 用新的修复提交向前修正。
- 不静默回退前一提交。

## 15. 完成声明规则

只有满足以下条件时，Agent 才能使用“已完成”：

- 任务包实现已提交。
- 必需测试和构建通过。
- Review 中没有未解决的 CRITICAL/HIGH 问题。
- 工作区状态和未提交变更归属已说明。
- 对应阶段退出门禁已核对。

否则必须使用“部分完成”“已实现但未验证”或“被阻塞”。
