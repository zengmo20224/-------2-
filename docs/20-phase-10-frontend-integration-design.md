# 第十阶段前端集成设计与实施计划

日期：2026-06-09

## 1. 文档目的

本文档用于约束第十阶段前端开发，明确小程序端、PC 管理后台、API 接入、测试、交付门禁和 Git 工作流。

本文档是阶段 10 的总体设计。D-010 已于 2026-06-09 决定采用“先管理后台，再小程序”。

后续细化阶段、依赖关系和任务包见 `docs/21-remaining-development-roadmap.md`，持续开发和 Agent 交接必须遵守 `docs/22-continuous-agent-development-rules.md`。

## 2. 阶段目标

第十阶段最终需要交付：

- 基于 `uni-app + Vue 3 + Pinia` 的微信小程序用户端。
- 基于 `Vue 3 + Vite + Element Plus` 的 PC 管理后台。
- 统一、类型明确、可测试的 API 请求层。
- 用户与管理员关键流程的 E2E 验证证据。
- 加载、空数据、错误、无权限和成功状态完整的页面体验。

第十阶段不能改变后端业务规则，也不能通过前端假数据伪造“流程已接通”。

## 3. 当前前端准备度审计

### 3.1 已具备能力

- 后端统一使用 `/api/v1` 前缀。
- 后端统一使用 `ApiResponse<T>` 成功和错误响应结构。
- 管理员登录、管理员身份查询和细粒度 RBAC 已实现。
- 服务目录、预约、社区、商品、自提订单、AI 边界和后台管理已有主要接口。
- 列表接口主要支持分页。
- 阶段 9 后台管理接口清单已记录在 `docs/19-phase-9-admin-api-handoff.md`。
- 后端全量自动化测试当前通过。

### 3.2 阻塞小程序真实流程的问题

以下问题必须在小程序身份相关流程接入前处理：

1. 微信登录接口当前只返回 `wechat_login_not_enabled`，不会创建用户或签发用户 Token。
2. 用户个人信息、宠物档案和地址管理没有 Controller API。
3. 预约、购物车、自提订单、发帖和用户 AI 接口都依赖当前用户身份。
4. 当前 Spring Security 除健康检查和登录接口外，要求所有 `/api/v1/**` 请求认证，因此服务目录、商品目录、话题和帖子列表也不能匿名浏览。
5. D-007 文件上传格式、数量、目录和压缩策略未决定，社区图片上传不能实现。
6. 真实 DeepSeek 未启用，AI 页面只能展示禁用或不可用状态，不能声称真实 AI 已接入。

### 3.3 阻塞完整管理后台的问题

- 服务分类和商品分类缺少后台新增、修改、禁用接口。
- 员工不可用时间缺少后台管理接口。
- 管理员账号和角色权限管理尚未实现。
- 经营数据仪表盘与导出接口尚未实现。
- 当前没有正式 OpenAPI 文档。
- 当前没有可复现的演示种子数据，门禁 9 尚未完全通过。

### 3.4 环境风险

- 管理后台开发环境应使用 Vite Proxy 转发 `/api`，避免依赖宽松 CORS。
- 小程序开发工具请求本地后端时，需要明确合法域名、本机代理或开发工具“不校验合法域名”设置，仅可用于本地开发。
- `mvn clean` 可能被本机 Java Language Server 占用 `target` 目录阻塞。
- JaCoCo 对 JSqlParser 存在既有仪器化警告，但后端测试命令可以成功退出。

## 4. D-010 实施顺序决策

### 方案 A：先管理后台，再小程序

优点：

- 管理员登录和权限链路已经可用。
- 阶段 9 后台基础资料 API 已完成。
- 可以先通过后台创建和维护服务、员工、排班、商品，再为小程序提供可用数据。
- 不依赖尚未启用的微信登录。

缺点：

- 用户核心业务价值展示较晚。
- 管理后台仍有分类管理、经营仪表盘等接口缺口。

### 方案 B：先小程序，再管理后台

优点：

- 优先展示预约、社区和到店自提等用户主线。
- 更贴近项目“微信小程序版本”的核心定位。

缺点：

- 当前没有可用的用户登录 Token。
- 用户、宠物、地址 API 缺失。
- 多个关键用户流程无法形成真实 E2E。

### 方案 C：两端按业务模块并行

优点：

- 可以围绕预约、订单、社区形成双端闭环。

缺点：

- 接口缺口、页面工作和联调问题会同时扩散。
- 对 AI Agent 协作、分支管理和回归测试要求最高。
- 在当前单人或少量 Agent 开发模式下容易产生范围失控。

### 已决定方案

用户已决定采用“方案 A：先管理后台，再小程序”。

原因：当前管理员认证与后台 API 的可用程度明显高于用户端。先完成管理后台，可以生成和维护小程序所需业务数据，并通过真实接口验证后台权限、预约处理、订单处理和内容审核。小程序实现前，再补齐用户认证、宠物和地址 API。

## 5. 总体前端架构

建议在仓库中建立独立前端工作区：

```text
frontend/
  admin-web/        # Vue 3 + Vite + Element Plus
  miniapp/          # uni-app + Vue 3 + Pinia
  shared/           # 可选，仅保存纯类型、状态映射和无平台依赖工具
```

约束：

- 不建立独立员工端。
- 不引入微前端。
- 不把小程序平台 API 放入 `shared/`。
- 不在前端保存后端密钥、数据库凭据或 DeepSeek API Key。
- 不在前端重新实现预约冲突、订单金额、内容风控或权限判定等后端业务规则。
- 共享代码只限于 API 类型、状态字典、日期金额格式化等无平台依赖逻辑。

### 6.3 管理员认证与权限

- 登录调用 `POST /api/v1/admin/auth/login`。
- 登录成功后保存管理员 Token，并调用 `GET /api/v1/admin/auth/me` 恢复会话。
- 请求头使用 `Authorization: Bearer <token>`。
- Token 不得写入 URL、日志或错误上报内容。
- 路由守卫负责“是否已登录”。
- 菜单和按钮根据权限码显示。
- 前端权限隐藏只改善体验，不能替代后端 `@PreAuthorize`。
- 后端返回 `401` 时清理会话并跳转登录页。
- 后端返回 `403` 时展示无权限页，不自动重试。

### 6.4 管理后台模块顺序

若用户批准管理后台优先，建议顺序：

1. 工程骨架、请求层、登录、会话恢复、权限路由。
2. 门店信息与门店配置。
3. 服务项目、员工、技能、排班。
4. 预约管理。
5. 商品与自提订单。
6. 社区审核、举报和敏感词。
7. 操作日志。
8. AI 分析和用量页面，仅展示后端实际可用状态。
9. 补齐缺失 API 后再实现分类、管理员角色和经营仪表盘。

## 8. API 请求层规范

两端都必须建立统一请求层，页面不能直接散落调用 Axios、`uni.request` 或拼接 URL。

请求层职责：

- 从环境配置读取 API Base URL。
- 自动注入对应 Token。
- 解析 `ApiResponse<T>`。
- 将分页数据统一映射为 `PageResponse<T>`。
- 对 `401`、`403`、`404`、`409`、`422` 和 `500` 统一处理。
- 支持请求取消，防止页面卸载后更新状态。
- 禁止自动重试写操作。
- 禁止把完整请求体、Token、密码或 AI 内容写入前端日志。

推荐类型：

```ts
interface ApiErrorDetail {
  field: string
  message: string
}

interface ApiError {
  code: string
  message: string
  details?: ApiErrorDetail[]
}

interface ApiResponse<T> {
  success: boolean
  data?: T
  error?: ApiError
  meta?: unknown
}

interface PageResponse<T> {
  items: T[]
  total: number
  page: number
  size: number
  totalPages: number
}
```

## 9. UI 状态与交互规范

每个数据页面必须覆盖：

- 首次加载状态
- 空数据状态
- 请求错误状态
- 无权限状态
- 提交中状态
- 提交成功状态
- 表单校验错误
- 状态冲突提示

写操作要求：

- 提交期间禁用重复提交。
- 删除、禁用、取消、拒绝、确认收款等高影响操作必须二次确认。
- `409 state_conflict` 必须提示用户刷新数据或重新选择时间。
- `422` 必须展示后端业务消息。
- 状态流转按钮只展示当前允许动作，但以后端校验结果为准。
- 金额统一按两位小数展示。
- 日期和时间格式统一，不依赖浏览器隐式时区转换。

## 10. 前端权限与安全规则

- 不在前端代码、`.env` 或构建产物中保存数据库密码、JWT 密钥、微信密钥或 DeepSeek API Key。
- 仅允许提交 `.env.example`，真实配置不得进入 Git。
- 禁止使用 `v-html` 渲染用户帖子或 AI 输出，除非经过可信净化。
- 用户输入在前端校验后仍必须由后端再次校验。
- 管理员权限码来自已认证会话，不允许本地手工提升权限。
- 错误提示不得展示 SQL、堆栈或 Provider 原始错误。
- 上传功能在 D-007 完整决定前禁止实现。

## 11. 状态字典管理

前端应建立集中状态字典，禁止各页面自行拼写状态字符串。

至少覆盖：

- 预约状态与支付状态
- 商品订单、自提和支付状态
- 内容审核状态
- 服务、员工、商品和门店状态
- AI 会话、报告与用量状态

状态字典只负责显示标签、颜色和允许展示的动作，不负责判断后端是否允许状态流转。

## 12. 测试策略

### 12.1 单元测试

必须覆盖：

- API 响应解析与错误映射
- Token 注入和 `401` 会话清理
- 权限码判断
- 状态字典
- 金额、日期和时间格式化
- 表单校验
- Pinia Store 状态变更

### 12.2 组件测试

必须覆盖：

- 加载、空数据、错误和成功状态
- 分页、筛选和表单提交
- 权限按钮隐藏
- 高风险操作二次确认
- `409`、`422` 和 `403` 提示

### 12.3 E2E 测试

管理后台最低 E2E：

1. 管理员登录并恢复会话。
2. 修改门店配置。
3. 创建员工、设置技能和排班。
4. 确认或拒绝预约。
5. 修改商品库存并处理自提订单。
6. 审核帖子并处理举报。
7. 查询操作日志。

小程序最低 E2E：

1. 用户登录。
2. 用户预约到店服务。
3. 超出服务半径的上门预约被拒绝。
4. 用户发布正常帖子。
5. 风险帖子进入审核或被拒绝。
6. 用户创建到店自提订单。
7. AI 客服基于种子门店、服务和 FAQ 回答。

真实用户认证、种子数据或 AI 未准备完成时，对应 E2E 必须标记为阻塞，不能用假数据宣称通过。

## 13. Git 与 Agent 执行规则

- 第十阶段代码只能在 `phase-10-frontend` 分支开发。
- 管理后台和小程序应使用独立提交，禁止一次提交混合两端大量变更。
- 每个功能按 RED、GREEN、重构提交检查点。
- 每个 Agent 开始前必须阅读 `AGENTS.md` 和本文档。
- 每个 Agent 只负责明确分配的前端模块，不能修改后端业务规则来绕过前端问题。
- 接口缺失时记录阻塞并提交接口需求，不允许前端静默伪造成功结果。
- 提交前必须运行对应 lint、类型检查、单元测试、构建和 `git diff --check`。

建议提交示例：

```text
docs: add phase 10 frontend integration design
test: add admin authentication store tests
feat: add admin authentication shell
test: add store management page tests
feat: add store management pages
```

## 14. 阶段拆解

### 10A：接入前门禁

交付物：

- 用户确认 D-010 实施顺序。
- 确定前端包管理器和 Node.js 版本。
- 明确用户认证测试方案。
- 补齐演示种子数据方案。
- 确认公开读取接口是否允许匿名访问。
- 建立准确的 API 接口清单。

退出标准：

- D-010 更新为已决定。
- 不存在影响首个前端子阶段的未决阻塞。
- 后端基线测试通过。

### 10B：前端基础设施

交付物：

- 前端工作区和应用骨架。
- 请求层、统一类型、错误处理。
- 路由、状态管理、环境配置。
- 基础布局和统一状态组件。
- lint、类型检查、单元测试和构建命令。

退出标准：

- 不包含业务假数据。
- 请求层测试通过。
- 两端构建通过，或只构建用户批准优先实现的一端。

### 10C：管理后台核心流程

仅在用户批准管理后台优先或并行方案时执行。

交付物：

- 登录与权限导航。
- 门店、服务、员工、排班。
- 预约、商品、自提订单。
- 社区审核、敏感词和操作日志。

退出标准：

- 管理后台关键 E2E 通过。
- 权限不足和状态冲突有验证证据。

### 10D：小程序核心流程

仅在用户批准小程序优先或并行方案，并解决用户认证与用户资料 API 后执行。

交付物：

- 用户身份、宠物和地址。
- 服务预约。
- 社区互动。
- 商品、购物车和自提订单。
- AI 页面与禁用状态。

退出标准：

- 小程序关键 E2E 通过。
- 不存在伪造用户身份或假接口结果。

### 10E：集成验收

交付物：

- 两端 API 联调清单。
- E2E 证据。
- 浏览器或小程序模拟器截图/录屏。
- 构建和测试报告。
- 已知风险与发布阻塞清单。

退出标准：

- UI 覆盖加载、空数据、错误和成功状态。
- 后端和前端构建测试通过。
- 门禁 9 和门禁 10 对照结果明确。
- Git 工作区干净并存在阶段提交。

2. 先编写请求层、Store 或组件测试，使测试处于 RED。
3. 实现最小功能使测试进入 GREEN。
4. 执行代码审查、安全检查和必要重构。
5. 运行 lint、类型检查、测试、构建和 `git diff --check`。
6. 提交独立 Git 检查点，并按 `AGENTS.md` 强制交接格式汇报。

禁止事项：

- 禁止因前端联调困难而放宽后端权限、绕过资源归属校验或修改状态流转规则。
- 禁止新增模拟生产登录、伪造用户 Token、伪造成功 API 或把静态假数据声明为联调完成。
- 禁止在同一提交中混入未批准的后端补缺、两端大规模页面和无关重构。
- 禁止在 D-007、D-008 未决部分上自行选择实现方案。
