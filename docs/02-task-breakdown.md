# 加速交付路线图

更新日期：2026-06-13

## 1. 当前基线

- 后端和管理端已有较多可复用实现。
- 用户端具备 UniApp H5 构建能力，但核心流程尚未全部真实联调。
- 商品订单和社区后端能力已有基础，应优先接通，而不是重写。
- 营销活动有基础数据模型，缺少完整可用流程。
- AI 保持禁用。
- 当前已知风险：认证过滤器对格式错误的 `Authorization` 请求头可能降级为匿名访问，必须在进入登录后 H5 流程前修复。

## 2. 交付顺序

### M0：定位与流程重置

目标：建立 H5 优先、少文档、纵向切片的单一执行基线。

完成条件：

- 核心文档完成合并。
- 旧阶段 brief、review 和 handoff 文档从工作区移除。
- Agent 以后不再创建同类文档。

### M1：H5 基础与可重复演示数据

目标：任何开发者都能启动后端和 H5，并看到稳定、可联调的数据。

任务：

- 修复认证请求头已知风险。
- 建立可重复执行的演示种子数据。
- 验证 H5 构建、路由、API 基址、加载态和错误态。
- 提供仅限开发/验收环境的测试身份入口。

退出条件：H5 可稳定启动，公开数据可浏览，登录后接口可联调。

### M2：公开浏览与营销

目标：未登录用户能完成主要内容浏览。

任务：

- H5 首页、服务、商品、社区和活动列表/详情。
- 基础搜索、分页或筛选。
- 营销活动后台维护和商品/服务关联。
- 公开 DTO 隐私验证。

退出条件：公开 H5 不依赖静态假数据，管理端发布的活动可在 H5 查看。

### M3：用户资料与真实预约

目标：测试身份可完成一条端到端预约。

任务：

- 个人资料、宠物、地址页面和真实 API。
- 服务选择、时段、距离校验、提交预约和预约查询。
- 管理端处理预约状态。
- 预约并发和状态流转验证。

退出条件：一次预约能从 H5 创建并在管理端正确处理。

### M4：真实商品与订单

目标：商品到订单的完整流程可真实使用。

任务：

- 商品详情、购物车、提交订单和订单查询。
- 服务端金额、商品快照、库存和幂等验证。
- 管理端处理订单状态。
- 关键订单流程 E2E。

退出条件：订单流程使用真实数据和事务规则，不依赖前端计算金额。

### M5：简化社区

目标：交付够用且安全的社区互动。

任务：

- 发帖、浏览、点赞、评论和收藏。
- 用户内容权限、公开隐私和基础审核。
- 移除或隐藏超出范围的复杂社交入口。

退出条件：核心互动可重复验证，用户之间无私有信息泄露。

### M6：发布收口

目标：形成可演示、可继续迭代的 H5 V1。

任务：

- ~~关键流程回归、响应式检查和错误体验修复。~~ 已完成
- ~~全量后端/前端测试、构建和关键 E2E。~~ 已完成
- ~~清理仅限开发环境的入口和测试数据风险。~~ 已完成
- 确认 H5 正式公开登录方案后再进入公开试运行。（阻塞于 P-001 未决事项）

## 3. 延后队列

- 微信登录和小程序适配。
- AI 功能。
- 在线支付、优惠券、积分、多门店和复杂营销。
- 非关键报表和大规模视觉重构。

## 4. 任务切片规则

- 每个 Agent 领取一个能独立演示和验证的纵向切片。
- 默认直接完成实现、测试、Review 修复和交接。
- 不为任务新增阶段计划文档；新的重要决定只更新现有核心文档。
- 先复用现有代码，再补缺口；发现旧实现有风险时在当前切片内修复。

## 5. 完成定义

一个切片只有同时满足以下条件才算完成：

- 用户流程真实可运行。
- 后端业务规则和权限生效。
- 相关自动化测试与构建通过。
- 关键失败路径有明确错误反馈。
- `git diff --check` 通过，未覆盖无关变更。
