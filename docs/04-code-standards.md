


# 代码与安全标准

## 1. 通用要求

- 只修改当前纵向切片需要的代码。
- 优先复用现有模式和工具，不做无关重构。
- 函数和模块保持单一职责，避免深层嵌套和超大文件。
- 对外输入必须校验，失败应返回稳定且可理解的错误。
- 不得静默吞掉异常，也不得向客户端泄露内部堆栈、SQL 或 Provider 错误。
- 不硬编码密钥、Token、密码和生产凭据。

## 2. 后端

- Controller 只处理协议、鉴权上下文和输入输出。
- Service 负责业务规则、权限检查和事务。
- Mapper 只负责数据访问。
- 金额使用精确数值类型，客户端金额只作为展示值。
- 订单、库存、预约和多表状态变化必须由服务端原子处理。
- 查询必须避免越权读取、批量无界读取和明显 N+1 问题。
- 对外响应优先使用 DTO，不直接暴露持久化实体。

## 3. 前端

- 用户端以 H5 响应式体验为第一目标，同时保持未来微信适配空间。
- 所有页面必须处理加载、空数据、错误和无权限状态。
- 禁止用静态数据掩盖尚未接入的真实接口。
- 身份和权限由后端决定；前端路由守卫和按钮隐藏只用于体验。
- H5 页面至少检查常见手机宽度和桌面浏览器布局。

## 4. 高风险业务

以下变更必须有直接测试和 Review：

- 认证、授权和用户数据隔离。
- 预约容量、距离、并发和状态机。
- 商品价格、库存、订单金额、幂等和状态机。
- 社区内容权限、审核和公开隐私。
- 管理端权限。

## 5. 当前范围限制

- 不新增 AI 业务依赖。
- 不新增在线支付、优惠券、积分、多门店或独立员工端。
- 营销活动不得直接绕过订单服务修改价格。
- 不为微信专属能力破坏 H5 核心流程。

## 6. 文档

- 只维护现有核心文档。
- 重要架构或产品决定更新 `docs/08-pending-decisions.md`。
- 不创建独立阶段 brief、review、handoff 或长篇过程报告。
- 代码已经清楚表达的细节不重复写入文档。
