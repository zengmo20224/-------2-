# 阶段 10H：前端设计稿与 GLM5.1 持续编码计划

## 1. 文档目的

本文档将已经确认的前端范围、Figma 设计稿、编码顺序、质量门禁和 Git 交接规则固化为 GLM5.1 可持续执行的任务包。

目标是让 GLM5.1 长时间稳定推进可验收的前端工作，而不是通过重复生成、无关重构或无效测试消耗 Token。

## 2. 开工门禁

GLM5.1 开始任一任务包前必须：

1. 完整阅读 `AGENTS.md` 规定的强制文档和本文档。
2. 运行 `git status --short --branch`、`git log --oneline -5`。
3. 确认当前任务依赖的 `10F-R2C` 至 `10F-R2E` API 契约修复已经通过门禁。
4. 不得覆盖当前工作区其他 Agent 的未提交修改。
5. 一次只领取一个任务包；完成测试、Review、验证和 Git 提交后才能开始下一个。

当前工作区存在并行修改。没有获得干净工作树或独立 worktree 前，禁止直接开始本计划的代码修改。

## 4. 明确边界

### 4.1 必须实现

- Vue 3、TypeScript、Element Plus 管理后台。
- 使用真实后端 API 和统一响应契约。
- Snowflake ID 在前端全程使用字符串。
- Loading、Empty、Error、403、并发冲突均为一等页面状态。
- 所有危险操作必须有二次确认和明确结果反馈。
- 文件上传前端限制为单文件不超过 `10MB`，同时依赖后端再次校验。
- 管理员权限控制到菜单、页面、操作按钮和 API 请求层。

### 4.2 禁止实现

- 不得使用假数据伪装接口已经可用。
- 不得自行增加多门店、微信支付、会员积分、优惠券或独立员工端。
- 不得提前启用 Redis。
- 不得自行实现微信登录，只能保留接口边界。
- 不得让 AI 直接访问数据库。
- 不得让 AI 提供宠物疾病诊断、药物处方或治疗承诺。
- 不得擅自修改后端状态名、字段名和 API。

### 4.3 阻塞能力的表现方式

- 经营分析 API 未提供：运营总览只展示真实接口可支持的工作队列和模块入口。
- 员工技能查询 API 未提供：技能编辑入口保持禁用，并说明阻塞原因。
- DeepSeek Provider 未启用：只实现安全的“功能尚未开放”状态，不调用模拟 AI。
- 小程序身份体系未完成：不进入小程序业务编码。

## 6. GLM5.1 任务包

建议分支命名：`phase-10h/<任务包编号>-<简短名称>`。

### H01：设计令牌与应用壳

- 将 Figma 令牌落到 CSS Variables 或项目现有主题入口。
- 重构管理后台侧边栏、顶栏、内容容器和响应式最小宽度。
- 保持现有路由与权限逻辑不变。
- 为菜单选中、折叠、403 跳转补测试。

验收：布局与设计稿一致；现有页面路由可进入；构建、类型检查、单测通过。

### H02：通用页面状态与反馈组件

- 实现 Loading、Empty、Error、403、阻塞能力提示。
- 实现统一成功、失败、重试和并发冲突反馈。
- 后端原始错误、SQL、堆栈和 Provider 错误不得直接显示。

验收：每个状态有组件测试；失败后可以重试；文案符合项目安全边界。

### H03：表格、筛选、抽屉和确认弹窗

- 实现通用表格外壳、筛选栏、详情抽屉和危险操作确认弹窗。
- ID、分页、筛选条件和操作按钮必须保持强类型。
- 抽屉关闭后清理临时状态，但不得错误清除列表筛选条件。

验收：组件测试覆盖打开、关闭、确认、取消、失败和权限不足。

### H04：登录与权限可见性

- 按设计稿完善登录页面。
- 菜单、页面和按钮根据细粒度管理员权限显示或禁用。
- 前端权限只负责交互可见性，后端仍是最终授权边界。

验收：登录失败不泄密；无权限访问进入 403；权限矩阵测试通过。

### H05：预约管理

- 实现筛选、分页、详情抽屉、确认、取消和员工重新分配。
- 预约 ID 必须按字符串处理。
- 对 `409` 或等价并发冲突响应显示“记录已更新，请刷新后重试”。
- 不得在前端自行推导或绕过后端状态机。

验收：预约关键流程集成测试和 E2E 通过；并发冲突、重复提交和权限不足均有测试。

### H06：员工与排班

- 实现员工列表、启停用、排班查看与维护。
- 员工技能查询 API 未提供时，技能入口保持禁用。
- 排班编辑必须展示服务时间冲突和保存失败状态。

验收：排班创建、修改、冲突、失败回滚的组件和 E2E 测试通过。

### H07：服务与商品管理

- 统一服务分类、服务项目、商品列表的筛选、状态和编辑体验。
- 文件上传限制为 `10MB`，类型与大小均需要前端校验。
- 商品价格、服务价格不得使用浮点误差敏感逻辑。

验收：金额显示、ID 字符串、上传限制和表单校验测试通过。

### H08：商品订单

- 实现订单筛选、详情、确认、取消和完成。
- 所有操作严格遵守后端订单状态机。
- 取消订单必须填写原因并二次确认。

验收：状态迁移、重复操作、非法迁移、取消原因和权限测试通过。

### H09：社区治理

- 实现帖子、评论、举报和敏感词的治理入口。
- 删除和处置动作必须说明影响并写入操作日志。
- 评论接口不可用时不得以本地假数据代替。

验收：举报处置和敏感词流程测试通过；危险操作有确认与失败反馈。

### H10：门店配置与操作日志

- 实现单门店信息配置和操作日志查询。
- 明确禁止多门店切换入口。
- 门店封面上传限制为 `10MB`。
- 操作日志只读，不允许前端提供修改或删除能力。

验收：保存失败保留输入；上传限制、权限和只读行为测试通过。

### H11：管理后台 E2E 与可访问性

- 使用 Playwright 覆盖登录、预约、订单、社区治理和门店配置关键流程。
- 检查键盘操作、焦点、表单标签、颜色对比和危险操作提示。
- 测试数据必须可重复初始化和清理。

验收：关键流程 E2E 全部通过；不得使用不稳定的固定等待。

### H12：管理后台质量门禁与交接

- 固化 lint、类型检查、单测、覆盖率、构建和 E2E 命令。
- 业务逻辑覆盖率不得低于 `80%`。
- 对照 Figma 截图检查核心页面视觉差异。
- 更新阶段状态、已知风险和未决事项。

验收：所有门禁命令通过；完成代码 Review 和安全 Review；提交可独立回滚。

## 7. 每个任务包的强制执行循环

1. `RED`：先添加失败测试，记录失败原因。
2. `GREEN`：只实现使测试通过的最小改动。
3. `IMPROVE`：清理重复、完善类型和错误处理。
4. Review：检查功能、权限、安全、可访问性和设计一致性。
5. Verify：运行任务包要求的单测、构建、E2E、`git diff --check`。
6. Commit：只提交本任务包文件，使用 Conventional Commits。
7. Handoff：按 `AGENTS.md` 强制交接格式报告。

## 8. Git 与并行开发规则

- 禁止直接在 `main` 实现。
- 每个任务包使用独立分支或独立 worktree。
- 禁止把其他 Agent 的未提交修改加入提交。
- 提交前必须执行 `git status --short` 和 `git diff --cached --stat`。
- 每个提交只能包含一个可独立验收的目的。
- 禁止使用 `git reset --hard`、`git clean -fd`、`git checkout -- .`。

建议提交示例：

```text
feat(frontend): implement petcare admin design tokens
test(frontend): cover booking conflict feedback
feat(frontend): implement booking detail drawer
docs(frontend): update phase 10h handoff
```

## 9. 最终验收门禁

管理后台达到以下条件后，才允许评估进入小程序编码：

- Figma 中的管理后台核心页面均已实现。
- 所有接口使用真实契约，不存在假数据兜底。
- ID 字符串契约、权限、预约并发冲突、订单状态机有直接测试。
- 业务逻辑覆盖率不低于 `80%`。
- lint、类型检查、单测、构建和关键 E2E 全部通过。
- 安全 Review 和代码 Review 无未解决的高风险问题。
- 已知阻塞项、未验证内容和回滚方式均已交接。
