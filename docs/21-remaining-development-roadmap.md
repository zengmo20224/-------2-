## 6. 阶段 10H：管理后台核心流程完成

目标：完成已具备后端 API 的管理后台核心流程。

执行顺序：

1. 服务项目管理。
2. 员工、技能和排班管理。
3. 预约管理。
4. 商品和库存管理。
5. 自提订单管理。
6. 帖子、评论、举报和敏感词管理。
7. 操作日志。
8. AI 分析报告和 AI 用量只读页面。

每个模块必须作为独立任务包开发：

```text
接口契约核对
-> RED 测试
-> 页面最小实现
-> 错误与权限状态
-> 组件或集成测试
-> 构建与 E2E
-> 独立 Git 提交
```

最低管理后台 E2E：

- 管理员登录和会话恢复。
- 修改门店配置。
- 创建员工、配置技能并设置排班。
- 确认或拒绝预约。
- 修改库存并处理自提订单。
- 审核帖子、评论并处理举报。
- 查询操作日志。

退出门禁：

- 所有已具备 API 的后台核心流程可用。
- `401`、`403`、`409`、`422` 和 `500` 有明确 UI 行为。
- 高影响写操作有二次确认和防重复提交。
- 后端权限仍是最终授权依据。
- 管理后台测试、构建和核心 E2E 通过。

## 13. 阻塞项处理规则

发现接口缺失、决策未完成或环境不可用时：

1. 停止依赖该阻塞项的实现。
2. 不得通过假数据、放宽鉴权或硬编码绕过。
3. 在交接中记录阻塞任务 ID、影响范围和已验证证据。
4. 可以继续执行不依赖该阻塞项的同阶段任务。
5. 阻塞解除后，从 RED 测试或可复现失败证据继续。

## 14. 阶段完成定义

一个阶段只有同时满足以下条件才算完成：

- 所有必做任务包完成。
- 退出门禁有命令或 E2E 证据。
- CRITICAL 和 HIGH Review 问题已解决。
- 文档与代码一致。
- 当前阶段变更已经提交 Git。
- 工作区中其他未提交变更已明确归属，不被误报为本阶段成果。
