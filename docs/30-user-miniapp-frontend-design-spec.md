# PetCare 用户端小程序前端设计规格

## 1. 文档定位

本文档是 PetCare O2O V1 用户端微信小程序的具体前端设计稿说明，用于指导后续产品确认、前端编码、接口补齐、测试和验收。

- Figma 文件：[PetCare O2O V1 前端设计稿与编码基线](https://www.figma.com/design/5rAoNS1QFy4He4IJhfNmMi)
- Figma 页面：`02 管理模块与小程序方向`
- 用户端设计区域：`用户端体验说明`、`用户端扩展流程说明`
- 设计基准：微信小程序，画板尺寸 `390 × 844`
- 推荐技术：`uni-app + Vue 3 + TypeScript + Pinia`

本文档描述完整的目标体验，但不代表所有功能当前均可编码或接入。任何依赖未完成 API 或未决事项的功能，都必须保留阻塞状态，不得使用假数据伪装可用。

## 2. 用户体验目标

用户端围绕三个核心目标设计：

1. 让用户快速理解门店能提供什么服务，并完成可信预约。
2. 让用户查看预约进度、浏览社区内容和提交到店自提订单。
3. 对尚未启用的微信登录、资料管理、图片上传和 AI 能力给出清晰状态，不制造虚假体验。

设计关键词：

- 温暖：视觉接近宠物照护场景，但不幼稚。
- 可信：价格、预约状态、付款方式和服务规则明确。
- 克制：不加入会员积分、优惠券、线上支付等 V1 范围外能力。
- 可恢复：预约冲突、地址超范围和接口不可用时提供下一步操作。

## 3. 产品与业务边界

### 3.1 V1 必须遵守

- 面向单体宠物门店，不展示门店切换。
- 用户提交预约后必须等待商家确认。
- 用户不能指定服务员工。
- 可预约时间由后端根据技能、排班、不可用时间和已有预约计算。
- 上门服务必须选择地址并校验服务半径。
- 商品只支持到店自提与线下付款。
- 所有外部 ID 在前端按字符串处理。
- AI 能力不可直接访问数据库。

### 3.2 V1 禁止出现

- 微信支付或其他线上支付。
- 多门店切换。
- 会员积分、优惠券、促销中心。
- 独立员工端入口。
- AI 疾病诊断、药物处方、替代兽医判断或治疗承诺。
- 前端自行计算预约冲突、最终价格、订单金额或服务半径。

### 3.3 当前阻塞能力

| 能力 | 当前状态 | 设计表现 |
|---|---|---|
| 微信登录 | 只保留接口边界 | 展示“微信登录尚未启用” |
| 用户资料、宠物、地址 API | 尚未补齐 | 对应入口显示 `API 待补齐` |
| 社区图片上传策略 | 尚未决定 | 发布页展示禁用上传区 |
| DeepSeek Provider | 尚未接入 | AI 页面展示功能未开放及安全边界 |
| 用户端真实写操作 | 依赖合法用户身份 | 登录不可用时禁止伪造 Token 或用户 ID |

## 4. 信息架构

底部导航固定为五个一级入口：

| 导航 | 核心职责 | 主要页面 |
|---|---|---|
| 首页 | 服务发现、门店动态、快捷预约 | 首页、服务列表、服务详情 |
| 预约 | 创建预约、查看状态、取消预约 | 三步预约、预约成功、我的预约、预约详情 |
| 社区 | 浏览、发布和互动 | 社区信息流、帖子详情、发布帖子 |
| 商品 | 浏览商品、购物车、自提订单 | 商品列表、商品详情、确认订单、我的订单 |
| 我的 | 用户身份、宠物、地址和个人记录 | 个人中心、宠物档案、地址、AI 状态 |

底部导航要求：

- 始终保持同一顺序。
- 当前入口使用主色和加粗文字。
- 二级页面可以隐藏底部导航，但必须提供明确返回方式。
- 不使用仅靠颜色区分的选中状态。

## 5. 核心用户流程

### 5.1 服务预约主流程

```text
首页
→ 服务列表
→ 服务详情
→ 选择宠物与服务方式
→ 选择可预约日期和时段
→ 确认预约信息
→ 提交预约
→ 等待商家确认
→ 我的预约查看状态
```

规则：

- 服务详情必须显示服务方式、预计时长、起始价格和预约规则。
- 创建预约第一步只能选择宠物、服务和服务方式，不能选择员工。
- 可预约时段必须来自后端。
- 提交时若发生并发冲突，保留已填写内容，只要求用户重新选择时间。
- 提交成功后的状态为“待商家确认”，不能显示为已确认。

### 5.2 上门服务流程

```text
选择上门服务
→ 选择服务地址
→ 后端校验服务半径
→ 地址有效：继续选择时间
→ 地址超范围：更换地址或改为到店服务
```

前端不得自行通过地图距离或固定公里数决定是否可服务。

### 5.3 商品自提订单流程

```text
商品列表
→ 商品详情
→ 加入购物车
→ 确认商品和数量
→ 提交到店自提订单
→ 等待门店确认库存
→ 到店线下付款和自提
```

订单确认页必须明确显示：

- 商品明细与商品合计。
- 到店自提。
- 到店后线下付款。
- 提交后由门店确认库存。

页面中禁止出现微信支付、立即支付、支付倒计时或线上付款二维码。

### 5.4 社区内容流程

```text
社区信息流
→ 帖子详情
→ 点赞 / 评论 / 收藏 / 举报

发布帖子
→ 选择话题
→ 输入正文
→ 选择图片（策略决定后）
→ 内容检查
→ 发布结果
```

阶段 14A 图片上传能力完成并通过门禁前，上传区域必须保持禁用，并说明原因。

## 6. 视觉设计系统

### 6.1 色彩

| Token | 色值 | 使用场景 |
|---|---|---|
| `--pc-user-primary` | `#11796F` | 主按钮、选中状态、关键链接 |
| `--pc-user-dark` | `#0C4D48` | 首页主视觉、强调区域 |
| `--pc-user-soft` | `#DFF2ED` | 标签、弱选中、成功状态 |
| `--pc-user-cream` | `#FAF8F3` | 页面背景 |
| `--pc-user-accent` | `#F5A623` | 首页主动作、重要提醒 |
| `--pc-user-accent-soft` | `#FFF0D1` | 待确认、规则提醒 |
| `--pc-user-danger` | `#D84D4D` | 错误、超范围、阻塞信息 |
| `--pc-user-ink` | `#19322E` | 主文字 |
| `--pc-user-muted` | `#71817D` | 辅助文字 |
| `--pc-user-line` | `#E2E9E6` | 输入框和分隔线 |

### 6.2 排版

- 页面主标题：`24px`，Bold。
- 卡片标题：`14px` 至 `18px`，Semi Bold 或 Bold。
- 正文：`12px` 至 `14px`。
- 辅助说明：`10px` 至 `11px`。
- 行高建议为字号的 `1.4` 至 `1.5` 倍。
- 中文界面实现时优先使用系统中文字体，避免强制加载外部字体。

### 6.3 间距与圆角

- 页面水平边距：`20px`。
- 卡片间距：`12px` 至 `16px`。
- 页面模块间距：`24px` 至 `32px`。
- 普通卡片圆角：`16px` 至 `18px`。
- 主视觉和重要状态卡片圆角：`20px` 至 `22px`。
- 按钮高度：`48px`。
- 底部导航高度：`64px`。

### 6.4 按钮层级

| 类型 | 视觉 | 使用规则 |
|---|---|---|
| 主按钮 | 主色背景、白色文字 | 每个页面最多一个主要动作 |
| 强调按钮 | 琥珀背景、深色文字 | 首页“开始预约”等业务入口 |
| 次按钮 | 浅绿色背景、深色文字 | 返回、切换服务方式 |
| 危险按钮 | 红色背景或红色文字 | 取消预约、举报确认 |
| 禁用按钮 | 低对比背景和说明 | API 或功能尚未启用 |

## 7. 页面设计规格

## 7.1 `U01 首页`

目标：让用户快速理解平台价值并开始预约。

页面结构：

1. 问候语与页面标题。
2. 预约主视觉卡片。
3. 常用服务快捷入口。
4. 门店动态。
5. AI 功能状态。
6. 底部导航。

关键文案：

- 主标题：`给它安心的照护时间`
- 说明：`洗护、美容与上门照护，提交后由门店确认`
- 主按钮：`开始预约`

状态要求：

- AI 未启用时显示“智能助手暂未开放”。
- 不展示虚假的热门服务数量、评分或折扣。

## 7.2 `U02 服务列表`

目标：区分到店和上门服务，帮助用户快速筛选。

页面结构：

- 全部、到店服务、上门服务筛选。
- 服务卡片：名称、预计时长、服务方式、起始价格。
- 加载、空列表、请求失败状态。

交互要求：

- 点击卡片进入服务详情。
- 切换筛选时保留滚动位置的策略需由实现阶段确认。
- 上门服务卡片标记“校验服务半径”。

## 7.3 `U03 服务详情`

目标：在用户预约前解释服务内容和规则。

必须显示：

- 服务名称、服务方式、预计时长。
- 起始价格及“实际价格由服务规格和宠物情况确定”说明。
- 服务包含内容。
- 商家确认规则。
- 用户不能指定员工。
- 线下付款说明。

主按钮：`选择宠物并预约`。

## 7.4 `U04 创建预约：宠物与服务方式`

目标：收集预约所需的基础信息。

字段：

- 选择宠物。
- 服务项目，只读或允许返回修改。
- 服务方式。
- 上门服务时选择地址。

必须显示：`员工由系统匹配`。

阻塞规则：

- 宠物 API 未完成时，不允许假设已有宠物。
- 微信登录未启用时，进入页面前展示身份能力未开放状态。

## 7.5 `U05 创建预约：选择时间`

目标：展示后端返回的实时可预约时段。

页面结构：

- 横向日期选择。
- 按上午、下午或晚上分组的可用时段。
- 实时变化提醒。
- 下一步按钮。

状态：

- 可用：可点击选择。
- 已选择：主色背景。
- 不可用：禁用并提供无障碍说明。
- 无时段：建议更换日期或服务。
- 请求失败：保留已选宠物和服务，支持重试。

## 7.6 `U06 创建预约：确认信息`

确认字段：

- 宠物。
- 服务及服务方式。
- 日期和时间。
- 员工：`由系统匹配`。
- 付款：`服务后线下付款`。
- 地址，仅上门服务显示。

提交前提示：门店确认前状态为“待确认”。

## 7.7 `U07 预约提交成功`

目标：明确告诉用户预约只是成功提交，尚未确认。

必须显示：

- 状态：`等待门店确认`。
- 预约编号。
- 服务、宠物和预约时间。
- 线下付款方式。
- `查看预约详情` 按钮。

## 7.8 `U08 预约时段冲突`

触发：提交时目标时段已经被其他有效预约占用。

文案要求：

- 标题：`需要重新选择时间`
- 说明：`可用时段会实时变化。其他预约信息已保留，只需重新选择时间。`

动作：

- 主动作：`重新选择时间`
- 次动作：`返回服务详情`

不得清空用户已经选择的宠物、服务方式和备注。

## 7.9 `U09 我的预约`

目标：统一查看预约状态和下一步动作。

筛选：

- 全部。
- 待确认。
- 进行中。
- 已完成。

预约卡片显示：

- 当前状态。
- 服务与宠物。
- 预约时间。
- 详情入口。

状态文案必须映射后端真实状态，不能由前端自行推导。

## 7.10 `U10 社区信息流`

页面结构：

- 推荐和话题筛选。
- 作者信息。
- 正文摘要。
- 图片区域。
- 点赞、评论、收藏和举报入口。

要求：

- 内容加载失败不得影响底部导航。
- 删除或审核中的帖子必须使用后端返回状态。
- 未登录用户是否可浏览取决于后端匿名读取决策。

## 7.11 `U11 发布帖子`

页面结构：

- 选择话题。
- 正文输入。
- 图片上传。
- 发布按钮。

校验：

- 正文长度。
- 敏感词或内容安全反馈。
- 图片数量、格式和大小，必须等待上传策略决定。

AI 发帖辅助未启用时不展示可点击入口。

## 7.12 `U12 商品列表`

页面结构：

- 商品分类筛选。
- 商品卡片：图片、名称、价格、库存状态。
- 购物车入口。

要求：

- 金额使用后端返回值和可靠金额格式化方法。
- 缺货商品不可加入购物车。
- 不展示优惠券、积分抵扣或线上支付活动。

## 7.13 `U13 确认自提订单`

必须显示：

- 商品及数量。
- 商品合计。
- 自提门店。
- 到店后线下付款。
- 门店确认库存说明。

主按钮：`提交自提订单`。

提交时不得调用线上支付能力。

## 7.14 `U14 个人中心`

页面结构：

- 用户身份卡片。
- 我的宠物。
- 我的地址。
- 我的预约。
- 自提订单。
- 收藏与互动。
- 联系客服。

当前阻塞状态：

- 微信登录未启用时显示“微信登录尚未启用”。
- 宠物与地址入口显示 `API 待补齐`，不可跳入假页面。

## 7.15 `U15 上门地址超范围`

触发：后端判断地址超出门店当前上门服务半径。

必须显示：

- 当前地址不可服务。
- 后端返回的距离或可理解说明。
- 当前服务半径。

动作：

- `更换服务地址`
- `改为到店服务`

不得在前端写死默认 `5km` 作为最终判断。

## 7.16 `U16 AI 能力未开放`

页面必须说明：

- 真实 Provider 启用后才开放。
- 未来可以提供门店规则、预约规则、商品自提规则和宠物日常陪伴。
- 不提供疾病诊断、药物处方、替代兽医判断或治疗承诺。
- 高风险症状必须建议联系宠物医院。

## 8. 通用页面状态

所有列表和详情页面必须实现以下状态：

| 状态 | 表现 |
|---|---|
| Loading | 使用骨架屏或局部加载，不阻塞整个应用 |
| Empty | 解释为什么为空并提供下一动作 |
| Error | 使用用户可理解文案，提供重试 |
| Offline | 提醒检查网络，保留本地表单输入 |
| Unauthorized | 说明需要微信登录，但不得伪造身份 |
| Forbidden | 不泄露资源是否存在 |
| Conflict | 保留用户输入，指导刷新或重新选择 |
| Blocked | 说明功能未启用或 API 待补齐 |

错误响应禁止直接展示：

- SQL。
- Java 堆栈。
- Provider 原始响应。
- Token、openid 或其他身份凭据。

## 9. 推荐组件拆分

```text
components/
  PcPageHeader
  PcBottomNavigation
  PcHeroCard
  PcServiceCard
  PcProductCard
  PcBookingCard
  PcStatusTag
  PcDateSelector
  PcTimeSlotGrid
  PcFormField
  PcPrimaryButton
  PcStatePanel
  PcBlockedFeature
  PcConfirmSheet
```

拆分原则：

- 页面负责业务编排，组件负责视觉和可复用交互。
- 请求不能散落在展示组件中。
- 不创建包含大量业务分支的“万能卡片”。
- 组件 Props 和事件必须有 TypeScript 类型。

## 10. 无障碍与交互要求

- 可点击区域建议不小于 `44 × 44px`。
- 状态不能只依靠颜色表达，必须有文字。
- 表单字段必须有常驻标签和校验信息。
- 错误后焦点应移动到错误摘要或首个错误字段。
- 弹窗、底部操作表和抽屉必须支持关闭和焦点恢复。
- 图片必须提供替代文本或明确标记为装饰图片。
- 尊重系统字体缩放，避免重要文案被截断。

## 11. 响应数据与前端状态

- 所有 Snowflake ID 使用 `string`。
- 金额避免使用浮点运算，展示值以 API 契约为准。
- 页面不得自行拼接或猜测后端业务状态。
- 请求层统一解析 `ApiResponse<T>` 和 `PageResponse<T>`。
- Token 只能来自合法身份流程。
- 表单提交必须防止重复点击。
- 预约冲突、订单非法状态迁移和资源归属错误必须分别处理。

## 12. 实施前置条件

在开始用户端真实编码前，至少确认：

1. 管理后台阶段门禁已通过。
2. 用户端认证方案已决定。
3. 用户、宠物和地址 API 已完成并有归属校验。
4. 服务、商品和社区公开读取策略已决定。
5. 预约与自提订单 API 契约可供用户端使用。
6. 图片上传策略已决定后，才实现社区图片上传。
7. AI Provider 真实启用后，才开放 AI 入口。

## 13. 设计验收清单

- [ ] 首页能清楚引导用户开始预约。
- [ ] 服务详情明确显示规则、方式、时长和线下付款。
- [ ] 用户端没有指定员工入口。
- [ ] 三步预约流程完整。
- [ ] 预约成功状态明确为等待商家确认。
- [ ] 并发冲突保留用户已经填写的信息。
- [ ] 上门地址超范围提供替代动作。
- [ ] 商品订单只包含到店自提和线下付款。
- [ ] 微信登录和缺失 API 使用真实阻塞状态。
- [ ] AI 页面完整表达安全边界。
- [ ] 所有页面具有 Loading、Empty、Error 和 Blocked 设计。
- [ ] 页面不包含 V1 范围外功能。

## 14. 后续编码顺序

本文档只定义设计规格，不批准立即进入用户端编码。满足实施前置条件后，建议按以下顺序执行：

1. 小程序工程骨架、主题、底部导航和统一状态页。
2. 服务、商品和社区公开浏览。
3. 用户身份、宠物与地址。
4. 三步预约、我的预约和冲突恢复。
5. 购物车和到店自提订单。
6. 社区发布与互动。
7. AI 未开放状态或真实 Provider 接入。
8. 用户端关键流程 E2E、可访问性和设计还原验收。
