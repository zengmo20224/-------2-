# 前期待决策事项

日期：2026-06-08

这些事项用于约束所有 AI Agent：未明确决定的内容不能自行选型或实现。

## 决策状态说明

- `未决`：禁止实现相关内容，必须先询问用户。
- `已决定`：可以按记录的决定实现。
- `部分决定`：只能实现已决定部分，其余内容继续等待用户确认。
- `暂缓`：当前版本不实现。

## D-001：后端认证框架

状态：已决定

决定日期：2026-06-08

决定：使用 Spring Security + JWT。

理由：采用标准企业级安全框架，便于展示认证、授权、方法级权限和测试能力。

影响：

- 登录流程
- Token 管理
- 权限注解
- 测试方式
- 后续微信登录接入

## D-002：主键生成策略

状态：已决定

决定日期：2026-06-08

决定：使用 MyBatis-Plus 雪花 ID，由应用层生成 `BIGINT` 主键；数据库表不使用 `AUTO_INCREMENT`。

影响：

- `schema.sql`
- 实体类配置
- 测试数据构造
- 后续分布式扩展能力

## D-007：文件上传限制

状态：已决定

决定日期：2026-06-08（部分决定），2026-06-12（全部决定）

已决定：

- 单文件大小上限为 10 MB。
- 图片格式白名单：`JPEG`、`PNG`、`WebP`。
- 每个帖子最多上传 6 张图片。
- 本地上传目录：`./uploads/community/`，按日期子目录组织（如 `./uploads/community/2026/06/12/`）。
- V1 不实现后端自动缩略图和图片压缩，上传时由前端压缩到合适尺寸。

理由：

- JPEG/PNG/WebP 覆盖主流格式，排除 GIF 降低内容审核负担。
- 6 张/帖符合主流社区惯例，平衡表达需求和存储成本。
- 本地文件系统足够 V1 单门店场景，V2 再考虑对象存储。
- 前端压缩减轻服务器负担，后端保留原始上传不做二次处理。

## D-008：AI Provider 接口协议

状态：已决定

决定日期：2026-06-08（部分决定），2026-06-12（全部决定）

已决定：

- 后续计划接入 DeepSeek。
- Provider Client 按 OpenAI 兼容的 Chat Completions 协议抽象。
- API Key 使用环境变量 `DEEPSEEK_API_KEY`。
- Base URL 使用可配置项，默认 `https://api.deepseek.com`。
- 默认模型：`deepseek-chat`。
- 超时：30 秒。
- 重试次数：1 次（最多 2 次调用，总计不超过 60 秒）。
- 最大 Token：2048。
- 用户调用频率限制：每用户每天 20 次。
- V1 不实现流式输出。

理由：

- `deepseek-chat` 是 DeepSeek 通用对话模型，性价比高，满足宠物门店场景。
- 30 秒超时适合国内网络环境。
- 1 次重试在失败时提供第二次机会，同时避免雪崩效应。
- 2048 Token 足够宠物门店咨询、发帖辅助等场景回复。
- 每用户每天 20 次防止滥用并控制成本。
- 流式输出增加前后端复杂度，V1 以简单稳定为主。

影响：

- AI Provider Client 接口
- 配置结构
- 用量日志
- 测试 Mock
- 限流方案

阶段 8 执行约束：

- `docs/17-phase-8-ai-provider-functions-plan.md` 将阶段 8 拆分为可立即实施的 8A 和等待决策的 8B。
- 8A 已完成 Provider 端口、禁用实现、Mock Provider、上下文、安全规则、持久化和聚合分析。
- 8B（真实 DeepSeek HTTP Client、超时、重试、最大 Token 和限流）可在阶段 14B 实施。
- 流式输出在 V1 不实现。

## D-011：雪花 ID 的 JSON 传输策略

状态：已决定

发现日期：2026-06-11

决定日期：2026-06-11

决定：

- 所有对外雪花 ID 字段序列化为 JSON 字符串。
- 前端统一使用字符串 ID。
- 仅处理 ID 字段，不全局转换所有 Java `Long`。
- `PageResponse.total`、数量、金额、库存、统计值和 Token 数等非 ID 数值保持 JSON 数字。

理由：

- 避免超过 JavaScript `Number.MAX_SAFE_INTEGER` 的雪花 ID 在浏览器中丢失精度。
- 避免全局 Long 序列化破坏分页和统计数值契约。

背景：

- 项目使用 MyBatis-Plus 雪花 `BIGINT` ID。
- 后端对外 DTO 广泛使用 Java `Long`。
- 管理后台当前将实体 ID 定义为 TypeScript `number`。
- 雪花 ID 可能超过 JavaScript `Number.MAX_SAFE_INTEGER`，浏览器解析 JSON 数字时可能丢失精度。

可选方案：

1. 所有对外雪花 ID 字段序列化为 JSON 字符串，前端统一使用 `string`；仅处理 ID 字段，不全局转换所有 Java `Long`。推荐。
2. 仅对指定 DTO 的 ID 字段使用字符串序列化。变更分散，容易遗漏。
3. 前端使用特殊 JSON BigInt 解析器。复杂度高，且默认 JSON 解析可能已丢失精度。
4. 限制 ID 小于 JavaScript 安全整数。与既定雪花 ID 策略冲突，不推荐。

实施约束：

- 后端必须先通过大 ID JSON 契约测试，再执行前端 `EntityId = string` 迁移。
- 禁止只修改前端 TypeScript 类型后声称已解决精度问题。
- 禁止将所有 Java `Long` 全局序列化为字符串，因为 `PageResponse.total` 等非 ID 数值必须保持数字。

详细计划见 `docs/24-phase-10f-r2-api-contract-cleanup-plan.md` 和 `docs/28-phase-10f-r2c-r2e-cross-layer-contract-plan.md`。

## D-012：单门店后台如何获得当前门店 ID

状态：已决定

发现日期：2026-06-11

决定日期：2026-06-11

决定：

- 新增受权限保护、由服务端解析唯一门店的 `/api/v1/admin/stores/current` 契约。
- 无门店时返回明确的资源不存在错误。
- 多于一个门店时返回明确的配置冲突错误。
- 管理后台不再使用固定 `STORE_ID = 1`，也不允许通过前端参数选择任意门店。

理由：

- 符合 V1 单门店边界。
- 避免固定 ID 与雪花 ID 策略冲突。
- 将资源范围校验保留在服务端，防止前端绕过。

背景：

- 管理后台当前固定使用 `STORE_ID = 1`。
- 固定值与雪花 ID 策略不兼容。
- 当前 `/api/v1/admin/auth/me` 不返回 `storeId`，也没有“当前门店”接口。

可选方案：

1. 新增受权限保护、由服务端解析唯一门店的 `/api/v1/admin/stores/current` 契约；无门店或多门店时明确失败。推荐。
2. 在 `/api/v1/admin/auth/me` 中返回 `storeId`，并明确管理员与门店关系。
3. 使用环境变量提供门店 ID，仅适合临时演示。
4. 保持固定 `STORE_ID = 1`。禁止。

实施约束：

- 旧 `{id}` 管理接口在兼容期必须验证目标 ID 等于服务端解析出的当前门店 ID。
- 禁止继续依赖固定 ID `1`。
- 禁止在多于一个门店时静默选择第一条记录。

详细计划见 `docs/24-phase-10f-r2-api-contract-cleanup-plan.md` 和 `docs/28-phase-10f-r2c-r2e-cross-layer-contract-plan.md`。

## D-013：用户端 E2E 认证方案

状态：已决定

发现日期：2026-06-12

决定日期：2026-06-12

决定：采用开发环境测试登录端点方案。

实现要求：

- 新增 `/api/v1/auth/dev-login`，接受手机号，查找或创建对应用户后签发 JWT。
- 仅在 Spring Profile `dev` 下启用，`prod` profile 下该端点不存在（不是返回 403，而是完全不可达）。
- 测试环境通过 `SPRING_PROFILES_ACTIVE=dev` 启用。
- 生产环境绝对不能暴露此端点。
- 开发登录不需要短信验证码，直接使用预设手机号。

理由：

- D-006 已决定 V1 只保留微信登录接口边界，不接入真实微信 AppID。
- 小程序用户端真实写操作（预约、下单、发帖）依赖合法用户身份。
- 开发环境测试登录端点允许 E2E 验证用户核心流程，不影响生产安全。
- 比实现短信验证码登录成本更低，不超出 V1 范围。

背景：

- 小程序预约、购物车、自提订单、发帖和用户 AI 接口都依赖当前用户身份。
- 没有合法用户身份就无法验证用户端核心业务流程。
- 微信登录接口当前只返回 `wechat_login_not_enabled`。

可选方案：

1. 开发环境测试登录端点。推荐。
2. 手机号+短信验证码登录。V1 范围外，需要短信服务商。
3. 邮箱/密码注册登录。V1 范围外，微信小程序内体验差。
4. 保持微信存根，所有用户写操作暂缓。小程序核心价值无法验证。

实施约束：

- 生产环境启动时必须有检测确保 dev-login 端点不可达。
- 禁止在非 dev 环境使用或测试该端点。
- 测试中使用的预设手机号必须在种子数据中定义。
- 详细计划见 `docs/31-phase-11-user-prerequisites-plan.md`。

## D-014：公开读取策略

状态：已决定

发现日期：2026-06-12

决定日期：2026-06-12

决定：采用内容全放行方案，所有公开资源的 GET 读取接口允许匿名访问，写操作和用户私有资源必须认证。

具体放行接口：

- `GET /api/v1/services/**`：服务目录、服务详情、服务分类。
- `GET /api/v1/products/**`：商品列表、商品详情、商品分类。
- `GET /api/v1/community/topics/**`：话题列表、话题详情。
- `GET /api/v1/community/posts/**`：帖子列表、帖子详情。
- `GET /api/v1/community/comments/**`：评论列表。

不允许匿名访问的接口：

- 所有 POST、PUT、PATCH、DELETE 操作。
- 用户私有资源（预约、订单、购物车、宠物、地址、收藏、点赞）。
- 管理后台所有接口。
- AI 相关接口。

理由：

- 与 `docs/30-user-miniapp-frontend-design-spec.md` 信息架构一致。
- 用户在小程序首页即可浏览服务、商品和社区内容，无需登录。
- 写操作和私有资源保持认证要求，安全可控。
- 服务端仍是最终授权方，放行仅限于 Spring Security 配置层。

背景：

- 当前 Spring Security 除健康检查和登录接口外，要求所有 `/api/v1/**` 请求认证。
- 服务目录、商品目录、话题和帖子列表也不能匿名浏览。
- 小程序首页无法展示任何内容，严重降低用户端价值。

可选方案：

1. 最小放行：仅服务目录、商品目录、话题和帖子列表。帖子详情无法浏览。
2. 内容全放行：所有公开资源 GET 读取。推荐。
3. 全部需认证：小程序首页无法展示内容。

实施约束：

- 放行接口不得泄露用户私有信息（如其他用户的预约、订单）。
- 帖子和评论列表只返回审核通过的内容。
- 管理后台接口不受影响。
- 详细计划见 `docs/31-phase-11-user-prerequisites-plan.md`。
