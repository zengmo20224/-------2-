# 前期待决策事项

日期：2026-06-08

这些事项用于约束所有 AI Agent：未明确决定的内容不能自行选型或实现。

## 决策状态说明

- `未决`：禁止实现相关内容，必须先询问用户。
- `已决定`：可以按记录的决定实现。
- `部分决定`：只能实现已决定部分，其余内容继续等待用户确认。
- `暂缓`：当前版本不实现。

## D-001：后端认证框架

状态：已决定

决定日期：2026-06-08

决定：使用 Spring Security + JWT。

理由：采用标准企业级安全框架，便于展示认证、授权、方法级权限和测试能力。

影响：

- 登录流程
- Token 管理
- 权限注解
- 测试方式
- 后续微信登录接入

## D-002：主键生成策略

状态：已决定

决定日期：2026-06-08

决定：使用 MyBatis-Plus 雪花 ID，由应用层生成 `BIGINT` 主键；数据库表不使用 `AUTO_INCREMENT`。

影响：

- `schema.sql`
- 实体类配置
- 测试数据构造
- 后续分布式扩展能力

## D-003：数据库版本迁移工具

状态：已决定

决定日期：2026-06-08

决定：当前项目仅维护 `schema.sql`，V1 不引入 Flyway 或 Liquibase。

影响：

- 阶段 1 输出就是完整初始化 SQL。
- 后续如果出现结构变更，先更新 `schema.sql` 和相关文档，不新增迁移工具。

## D-004：预约并发防冲突方案

状态：已决定

决定日期：2026-06-08

决定：使用“员工日期锁定点 + 数据库事务 + 冲突二次检查 + 死锁有限重试”，不使用 Redis。

用户已批准在 `schema.sql` 中新增 `staff_booking_lock` 辅助表。详细实现要求见 `docs/09-booking-concurrency-control.md`。

## D-005：管理员权限粒度

状态：已决定

决定日期：2026-06-08

决定：使用细粒度 RBAC。`admin_user.role` 保存基础角色编码，新增角色、权限、角色权限关系表；Spring Security 使用权限码进行方法级授权。

初始权限矩阵和权限码规范见 `docs/10-admin-permission-design.md`。

## D-006：微信登录开发模式

状态：已决定

决定日期：2026-06-08

决定：V1 当前只保留微信登录接口和适配器边界，不接入真实微信 AppID，不实现模拟登录。

实现要求：

- 后端可以定义 `WechatLoginProvider` 或等效接口。
- Controller 可以保留登录入口，但真实 code2session 调用返回“暂未启用”或 feature disabled。
- 不允许伪造真实 openid 作为生产逻辑。

## D-007：文件上传限制

状态：部分决定

决定日期：2026-06-08

已决定：单文件大小上限为 10 MB。

仍未决定：

- 本地上传目录。
- 是否需要缩略图和图片压缩。

## D-008：AI Provider 接口协议

状态：部分决定

决定日期：2026-06-08

已决定：

- 后续计划接入 DeepSeek。
- Provider Client 按 OpenAI 兼容的 Chat Completions 协议抽象。
- API Key 使用环境变量 `DEEPSEEK_API_KEY`。
- Base URL 使用可配置项，默认 `https://api.deepseek.com`。

仍未决定：

- 默认模型名称。
- 超时、重试次数、最大 Token。
- 用户和管理员调用频率限制。
- 是否需要流式输出。

影响：

- AI Provider Client 接口
- 配置结构
- 用量日志
- 测试 Mock
- 限流方案

阶段 8 执行约束：

- `docs/17-phase-8-ai-provider-functions-plan.md` 将阶段 8 拆分为可立即实施的 8A 和等待决策的 8B。
- 8A 可以实现 Provider 端口、禁用实现、Mock Provider、上下文、安全规则、持久化和聚合分析。
- 8B 的真实 DeepSeek HTTP Client、超时、重试、最大 Token、限流和流式输出必须等待本决策补全。
- 在本决策补全前，不能声称真实 DeepSeek 已接入或生产 AI 调用已可用。

## D-009：V1 是否启用 Redis

状态：已决定

决定日期：2026-06-08

决定：

- V1 不启用 Redis。
- 预约冲突防护优先使用数据库事务。
- 只有明确出现缓存、共享限流或分布式锁需求时才重新评估。

1. 将状态改为“已决定”。
2. 写明选择结果、日期和理由。
3. 更新受影响的架构、任务和实施计划文档。
4. 提交独立 Git 文档提交。
5. 再开始相关代码实现。
