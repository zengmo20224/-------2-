# 风险管理实验报告

## 一、实验要求

根据个人在完成项目的过程中的实际情况，对风险进行分析，制定风险计划。

## 二、实验内容

### 1、项目名称

**PetCare O2O —— AI 增强宠物门店 O2O 服务预约与客户运营平台**

### 2、负责模块

后端服务预约模块（booking）、社区模块（community）、管理后台前端（admin-web）中与预约管理、社区审核相关的页面。

具体包括：

- 后端：`BookingApplicationServiceImpl`（预约核心业务逻辑）、`CommunityPostApplicationService`（社区帖文与审核逻辑）
- 前端：管理后台的预约管理页面（`/bookings`）、社区审核页面（`/community/posts`、`/community/reports`）
- 跨层：前端请求层（`request.ts`）与管理员认证状态管理（`store/user.ts`）

### 3、风险识别方法

本项目采用 **多种风险识别方法组合** 的方式：

| 方法 | 说明 | 应用场景 |
|------|------|----------|
| **头脑风暴法** | 结合个人开发经验，对项目中可能出现的问题进行发散式列举 | 识别技术选型、开发流程、外部依赖等方面的风险 |
| **检查表法（Checklist）** | 参照项目 `AGENTS.md` 和 `docs/20-phase-10-frontend-integration-design.md` 中的集成门禁清单，逐项排查 | 识别代码安全、接口契约、测试覆盖等质量风险 |
| **文档审查法** | 阅读需求文档（`requirements-source.md`）、架构设计（`01-architecture-design.md`）和待决策清单（`08-pending-decisions.md`），发现矛盾和缺口 | 识别需求不明确、接口缺失、未决事项等 |
| **历史经验法** | 参考项目 Git 提交历史和阶段文档中记录的实际阻塞问题 | 识别环境配置、工具链兼容、构建失败等已发生的风险 |

**识别过程举例**：

通过文档审查法阅读 `docs/20-phase-10-frontend-integration-design.md` 第 3.2 节，发现"微信登录接口当前只返回 `wechat_login_not_enabled`"，识别出用户认证缺失风险；通过检查表法对照 `request.ts` 的响应拦截器，发现 401/403 处理逻辑已覆盖，但网络超时重试策略缺失，识别出健壮性风险。

### 4、风险登记册

| 序号 | 类别 | 风险描述 | 概率 | 影响 | 风险等级 | 风险应对措施 | 责任人 |
|:---:|------|----------|:---:|:---:|:---:|----------|:---:|
| 1 | 技术风险 | **微信小程序用户认证缺失**：后端微信登录接口仅返回 `wechat_login_not_enabled`，不创建用户也不签发 Token，导致小程序端所有依赖用户身份的功能（预约、购物车、发帖、AI 对话）无法形成真实 E2E 闭环 | 高 | 高 | 🔴 高 | ① 采用"先管理后台、再小程序"的实施顺序（D-010 已决策），规避当前阻塞；② 在小程序开发前优先补齐用户认证 API；③ 若需测试，必须由项目负责人批准仅测试环境可用的认证入口，禁止伪造 Token | 曾同学 |
| 2 | 技术风险 | **前后端接口契约不一致**：后端 DTO 字段与前端 TypeScript 类型定义可能出现不匹配。例如后端 `BookingResponse` 的时间字段格式、分页字段命名（`items` vs `records`）可能与前端 `ApiResponse<T>` / `PageResponse<T>` 类型定义不一致，导致联调时数据解析失败 | 高 | 中 | 🟠 高 | ① 前端请求层（`request.ts`）统一使用 `ApiResponse<T>` 封装，强制类型约束；② 每个模块开发前先基于真实 Controller 和 DTO 建立接口契约清单；③ 提交前运行 TypeScript 类型检查（`tsc --noEmit`）和后端测试 | 曾同学 |
| 3 | 安全风险 | **管理员 Token 泄露与 XSS 攻击**：前端 Token 存储在 `localStorage`（见 `store/user.ts` 第 14 行），一旦存在 XSS 漏洞即可被盗取；社区帖文如果使用 `v-html` 渲染用户内容，存在存储型 XSS 风险 | 中 | 高 | 🟠 高 | ① 项目规范已明确禁止使用 `v-html` 渲染用户帖子和 AI 输出；② 后端内容审核服务（`ContentModerationService`）对所有用户提交内容进行敏感词检测；③ Token 有效期设置合理，401 时自动清理并跳转登录页；④ 后端使用 `@PreAuthorize` 进行权限校验，不依赖前端权限隐藏 | 曾同学 |
| 4 | 进度风险 | **后端 API 缺口阻塞前端开发**：多个管理后台页面依赖的后端接口尚未实现，包括：服务分类/商品分类的增删改接口、员工不可用时间管理接口、管理员账号与角色权限管理接口、经营数据仪表盘接口（参见设计文档第 3.3 节）。这些缺口会导致前端页面只能展示骨架，无法完成真实联调 | 高 | 中 | 🟠 高 | ① 按设计文档 6.4 节的模块顺序，优先开发已有后端 API 支撑的页面；② 接口缺失时记录阻塞并提交接口需求，禁止前端静默伪造成功结果；③ 分阶段验收，每个阶段门禁通过后再进入下一阶段 | 曾同学 |
| 5 | 技术风险 | **预约并发冲突与状态流转错误**：预约模块（`BookingApplicationServiceImpl`）涉及多表状态变更（预约状态、员工排班、服务项目状态），高并发场景下可能出现同一时段重复预约、状态流转不符合状态机规则（`BookingStateMachine`）的问题 | 中 | 高 | 🟠 高 | ① 后端预约状态流转使用事务（`@Transactional`）保证多表一致性；② 采用领域状态机（`BookingStateMachine`）严格控制状态流转；③ 前端收到 409 状态冲突响应时提示用户刷新数据（已在 `request.ts` 第 55 行处理）；④ 预约模块必须有直接单元测试覆盖 | 曾同学 |
| 6 | 环境风险 | **开发环境构建与工具链冲突**：项目文档记录了多个环境问题——`mvn clean` 可能被 Java Language Server 占用 `target` 目录阻塞、JaCoCo 对 JSqlParser 存在仪器化警告、前端需要 Vite Proxy 转发避免 CORS 问题、小程序开发工具需要配置本地代理或不校验合法域名 | 中 | 中 | 🟡 中 | ① 开发前确认 Node.js 和 Java 版本基线；② 后端构建前关闭占用进程；③ 管理后台通过 Vite Proxy 配置 `/api` 转发，避免依赖宽松 CORS；④ 环境配置写在 `.env.example`，真实配置不进入 Git | 曾同学 |
| 7 | 质量风险 | **测试覆盖率不足**：项目要求业务逻辑测试覆盖率不低于 80%，但前端管理后台刚进入阶段 10，单元测试和 E2E 测试尚未完整建立。预约排班、距离校验、状态流转、内容风控、订单金额等核心逻辑如果测试缺失，回归时容易引入缺陷 | 中 | 中 | 🟡 中 | ① 严格执行 TDD 流程（RED → GREEN → 重构）；② 后端预约、订单、风控等关键模块必须编写单元测试（`BookingStateMachine`、`BookingDistanceCalculator` 等）；③ 管理后台最低 E2E 覆盖：登录→修改门店→创建员工排班→确认预约→处理订单→审核帖子；④ 使用 JaCoCo 监控覆盖率 | 曾同学 |
| 8 | 需求风险 | **已决定能力被提前或越界实现**：D-007、D-008、D-013、D-014 已于 2026-06-12 决定，但文件上传、真实 DeepSeek、测试登录和匿名读取仍必须按对应阶段实施，不能把“已决定”误解为允许跨阶段开发 | 中 | 中 | 🟡 中 | ① 严格遵守阶段门禁；② 测试登录仅允许在 `test` Profile；③ 上传和真实 DeepSeek 只在阶段 14 独立实施；④ 每项能力完成后执行安全和失败场景测试 | 曾同学 |
| 9 | 管理风险 | **单人开发模式下分支与代码管理混乱**：项目使用多分支开发（当前在 `phase-10-frontend`），单人配合 AI Agent 开发时，容易出现：一次提交混合多模块变更、覆盖其他 Agent 已有变更、跳过阶段门禁直接开发后续功能等问题 | 中 | 中 | 🟡 中 | ① 严格遵守 Git 安全工作流：每个功能独立提交，禁止混合多端变更；② 修改前必须运行 `git status` 和 `git log` 检查工作区状态；③ 每个阶段完成后对照集成门禁（`docs/07-integration-gates.md`）检查；④ 禁止使用 `git reset --hard`、`git clean -fd` 等破坏性命令 | 曾同学 |
| 10 | 外部依赖风险 | **AI 服务与外部 API 可用性不确定**：AI 模块（`AiConversationApplicationServiceImpl`）依赖外部 AI Provider，真实 DeepSeek 尚未启用。如果 AI Provider 不可用或 API 额度用尽，AI 客服、发帖辅助、经营分析等功能将无法运行 | 中 | 低 | 🟢 低 | ① AI 页面仅按后端真实启用状态展示，未启用时显示禁用或不可用提示，不伪造 AI 已接入；② AI 模块使用统一 Provider 适配器模式，便于切换；③ 核心业务（预约、订单、社区）不依赖 AI 功能，AI 为增值功能 | 曾同学 |

## 五、总结

本项目共识别出 **10 项风险**，其中：

- 🔴 **高风险**：1 项（用户认证缺失）
- 🟠 **较高风险**：4 项（接口契约不一致、Token 安全、后端 API 缺口、预约并发冲突）
- 🟡 **中等风险**：4 项（环境冲突、测试覆盖、未决决策、分支管理）
- 🟢 **低风险**：1 项（AI 服务可用性）

针对高风险和较高风险项，已制定了具体的规避、转移和缓解措施。核心策略是：**先完成管理后台（已有后端 API 支撑），再补齐小程序依赖的用户认证与资料 API**，同时严格执行 TDD 和阶段门禁，确保每个阶段的交付质量。
